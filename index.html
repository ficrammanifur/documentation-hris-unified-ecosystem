System Overview

A corporate HRIS (Human Resource Information System) designed for a single company with mobile attendance, admin management, and API-first architecture.

Flutter Mobile App

Employee-facing mobile app for daily attendance with photo capture and GPS validation.

  • Photo + GPS attendance
  • Leave requests
  • Profile management
  • Push notifications

React Admin Panel

Web dashboard for HR admins and managers to manage employees and view analytics.

  • Employee CRUD
  • Attendance reports
  • Dashboard analytics
  • Role-based views

Laravel API Backend

RESTful API backend handling authentication, data, and business logic.

  • Sanctum token auth
  • Role-based access
  • API-first design
  • Anti-cheat logic

API-First Architecture

Controller: Handles HTTP request/response only. Delegates to Services.
Service: Contains business logic (attendance anti-cheat, leave approval rules).
Repository: Optional data access abstraction for complex queries.
Request: Validates incoming data before reaching the controller.
Resource: Transforms Eloquent models into consistent JSON responses.

Key Relationships

  • users -- roles: Many-to-One. Each user has one role (admin, hr, manager, employee).
  • employees -- users: One-to-One. Each employee links to a user account for authentication.
  • employees -- departments: Many-to-One. Each employee belongs to one department.
  • departments.head_id -- employees: Self-referencing FK. Department head is an employee.
  • attendances -- employees: Many-to-One. Each attendance record links to one employee.
  • leaves -- employees: Many-to-One. Leave requests reference the requesting employee.
  • leaves.approved_by -- users: FK to users table for the approving manager/HR.

Anti-Cheat Rules

1. One Check-In Per Day

A UNIQUE composite index on employee_id + date prevents duplicate check-ins at the database level.

// AttendanceService.php
public function checkIn(Employee $employee, array $data): Attendance
{
    // DB constraint: UNIQUE(employee_id, date)
    $existing = Attendance::where('employee_id', $employee->id)
        ->whereDate('date', today())
        ->first();

    if ($existing) {
        throw new AlreadyCheckedInException(
            'You have already checked in today.'
        );
    }

    return Attendance::create([...]);
}

2. Location Radius Validation

GPS coordinates are validated against the office location using the Haversine formula. Default radius: 200 meters.

// config/attendance.php
return [
    'office_latitude' => env('OFFICE_LAT', -6.2088),
    'office_longitude' => env('OFFICE_LNG', 106.8456),
    'max_radius_meters' => env('ATTENDANCE_RADIUS', 200),
];

// AttendanceService.php
private function validateLocation(float $lat, float $lng): bool
{
    $distance = $this->haversineDistance(
        $lat, $lng,
        config('attendance.office_latitude'),
        config('attendance.office_longitude')
    );

    return $distance <= config('attendance.max_radius_meters');
}

3. Duplicate Prevention

Rate limiting middleware prevents rapid successive requests. Combined with the unique DB constraint for bulletproof protection.

// AttendanceThrottle middleware
Route::middleware(['auth:sanctum', 'throttle:attendance'])
    ->group(function () {
        Route::post('/check-in', [AttendanceController::class, 'checkIn']);
        Route::post('/check-out', [AttendanceController::class, 'checkOut']);
    });

// RouteServiceProvider
RateLimiter::for('attendance', function (Request $request) {
    return Limit::perMinute(3)->by($request->user()->id);
});

4. GPS Spoofing Mitigation

Basic client-side checks combined with server-side anomaly detection for mock location prevention.

// Flutter: Check for mock location
import 'package:geolocator/geolocator.dart';

Future isMockLocation(Position position) async {
  if (Platform.isAndroid) {
    return position.isMocked;
  }
  return false;
}

5. Logout revokes token on server

Security Structure

Multi-layered security with Sanctum authentication, role middleware, input validation, and file upload protection.

1. Sanctum Token Authentication

Laravel Sanctum provides lightweight API token authentication. Tokens are stored hashed in the database and sent via Bearer header.

// Login flow
public function login(LoginRequest $request)
{
    $user = User::where('email', $request->email)->first();

    if (!$user || !Hash::check($request->password, $user->password)) {
        throw ValidationException::withMessages([
            'email' => ['Invalid credentials.'],
        ]);
    }

    $token = $user->createToken('mobile-app')->plainTextToken;

    return response()->json([
        'success' => true,
        'data' => [
            'user' => new UserResource($user),
            'token' => $token,
        ],
    ]);
}

2. Role-Based Middleware

Custom middleware checks user roles before allowing access to sensitive endpoints.

// RoleMiddleware.php
class RoleMiddleware
{
    public function handle(Request $request, Closure $next, ...$roles)
    {
        $user = $request->user();

        if (!$user || !in_array($user->role->name, $roles)) {
            return response()->json([
                'success' => false,
                'message' => 'Unauthorized. Insufficient permissions.'
            ], 403);
        }

        return $next($request);
    }
}

class StoreEmployeeRequest extends FormRequest
